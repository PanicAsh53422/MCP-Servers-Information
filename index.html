Section 01
What is MCP?
Model Context Protocol is an open standard by Anthropic that lets AI models communicate with external tools, APIs, and data sources through one unified interface. Think of it as USB-C for AI integrations.
01
Why does MCP exist?
▶
Before MCP, every AI tool integration was a custom one-off build. MCP standardizes how models discover and invoke tools — so a server written once works with any MCP-compatible client: Claude Desktop, Cursor, VS Code, Continue, and more.
02
How is it different from a REST API?
▶
REST APIs are designed for app-to-app communication with a known schema. MCP is designed for AI-to-tool communication — it includes a built-in discovery step where the model queries what tools are available at runtime, then decides which to call based on the user's request.
03
Who created it and who supports it?
▶
Created by Anthropic and open-sourced in late 2024. Now supported across Claude Desktop, Cursor, VS Code Copilot Chat, Continue, Zed, and a growing list of AI clients. The spec and SDKs are maintained at github.com/modelcontextprotocol.
Section 02
How it Works
MCP uses a client-server architecture. The AI client connects to one or more servers, each of which exposes capabilities the model can use during a conversation.
01
The three layers: Host, Client, Server
▶
Host — the application the user runs (e.g. Claude Desktop, Cursor). It manages the user session and holds one or more clients.

Client — lives inside the host; maintains a 1:1 connection with one MCP server. Forwards tool calls from the model to the server.

Server — a separate process (or remote service) that exposes tools, resources, and prompts. You build this.
02
Transport types: stdio vs HTTP+SSE
▶
stdio — the host spawns your server as a child process and communicates over stdin/stdout. Used for local servers. Simple, no networking required.

HTTP + SSE — server runs as an HTTP service; client connects via Server-Sent Events for streaming. Used for remote or shared servers.
03
The request flow
▶
1. User sends a message to the AI.
2. Model decides a tool call is needed and emits a tool-call request.
3. Client routes the request to the appropriate MCP server.
4. Server executes the tool and returns a result.
5. Result is injected back into the model's context.
6. Model generates a final response using the result.
Section 03
Core Capabilities
MCP servers expose three primitives. Tools are the most common — resources and prompts round out the spec for more advanced use cases.
01
Tools — functions the model can call
▶
A tool is a named function with a description and a JSON Schema defining its inputs. The model reads the description to decide when to use it, then passes validated arguments. Tools can do anything: query a database, call an API, run a shell command, write a file.
02
Resources — read-only context the model can access
▶
Resources are identified by a URI (e.g. file:///path/to/doc.txt) and return text or binary content. Unlike tools, resources are passive — the model or host fetches them, rather than invoking them with arguments. Useful for exposing files, database rows, or documentation.
03
Prompts — reusable prompt templates
▶
A server can expose named prompt templates with dynamic arguments. The host can surface these as slash-commands or quick actions in the UI. Less commonly used than tools, but useful for standardizing repeated workflows — e.g. a "summarize PR" prompt that always fetches the latest diff first.
Section 04
Popular Servers
The MCP ecosystem has hundreds of servers. These seven are the most useful for classroom work — each installs with a single command and covers a distinct capability.
01
Filesystem — read and write local files
▶
The simplest MCP server. Gives Claude access to a directory you specify — it can list, read, write, and search files within it. Great starting point for understanding how MCP tools work.

Classroom use: Have Claude read your notes folder and summarize what you worked on this week.

npx -y @modelcontextprotocol/server-filesystem /path/to/folder
02
GitHub — repos, PRs, issues, and code
▶
Search repositories, read file contents, list and create pull requests, open issues, and post comments — all without leaving Claude. Requires a GitHub personal access token.

Classroom use: Ask Claude to review your latest PR diff and suggest improvements before you merge.

npx -y @modelcontextprotocol/server-github
03
Postgres — SQL queries against a live database
▶
Runs read-only SQL queries against a PostgreSQL database and returns results as plain text. Claude can inspect schema, write and explain queries, and translate business questions into SQL.

Classroom use: Hand Claude an unfamiliar schema and ask it to explain each table in plain English.

npx -y @modelcontextprotocol/server-postgres postgresql://localhost/mydb
04
Brave Search — real-time web results
▶
Queries the Brave Search API for live web results. Claude gets titles, URLs, and descriptions — then synthesizes them into a structured answer. Requires a free Brave API key.

Classroom use: Research a topic and have Claude synthesize results from multiple sources in one response.

npx -y @modelcontextprotocol/server-brave-search
05
Puppeteer — browser automation and scraping
▶
Controls a headless Chrome instance. Claude can navigate pages, click elements, wait for dynamic content, take screenshots, and extract structured data from any website.

Classroom use: Scrape a product page for pricing data, or screenshot a dashboard for a weekly report.

npx -y @modelcontextprotocol/server-puppeteer
06
Slack — messages, channels, and users
▶
Send and read messages, list channels, look up users, and post to Slack workspaces. Requires a Slack bot token and team ID set as environment variables.

Classroom use: Have Claude monitor a support channel and draft responses to common questions.

npx -y @modelcontextprotocol/server-slack
07
Fetch — retrieve content from any URL
▶
Fetches raw HTML or text from any public URL without opening a browser. Useful for pulling in documentation pages, RSS feeds, or plain-text files directly into your conversation.

Classroom use: Point Claude at an API docs page and ask it to write a client based on what it reads.

npx -y @modelcontextprotocol/server-fetch
Section 05
Use Cases
MCP servers become powerful when combined. These scenarios show how pairing the right servers with Claude solves real problems you'd otherwise solve manually.
01
Research Assistant
▶
Problem: You need to research a topic across multiple sources and synthesize findings without copy-pasting from tab to tab.

Servers: Brave Search + Fetch

Outcome: Claude searches the web for your query, fetches the full text of the top results, and returns a structured summary with citations — all in one response.
02
Code Reviewer
▶
Problem: You want AI feedback on a pull request without pasting the whole diff into a chat window.

Server: GitHub

Outcome: Claude fetches the PR diff directly from GitHub, reads the changed files, and returns structured review comments organized by file — pointing to specific lines.
03
Database Explorer
▶
Problem: A teammate handed you a Postgres database with no documentation and you need to understand it fast.

Server: Postgres

Outcome: Claude queries the schema, describes each table in plain English, maps out the relationships, and answers questions like "which table tracks user sessions?" without you writing a single query.
04
Web Scraper
▶
Problem: You need structured data from a website that doesn't offer an API.

Server: Puppeteer

Outcome: Claude navigates to the page, waits for dynamic content to load, extracts the data you describe (prices, names, dates), and returns it as a clean table ready to copy into a spreadsheet.
05
Team Automator
▶
Problem: New GitHub issues pile up and it takes time to triage them, assign labels, and notify the team.

Servers: GitHub + Slack

Outcome: Claude reads open issues, categorizes them by type (bug, feature, question), posts a triage summary to a Slack channel, and suggests label assignments — in one prompt.
Section 06
Getting Started
Connect an MCP server to Claude Desktop in five steps. We'll use the Filesystem server as the example — it requires no API key and works immediately.
1
Download Claude Desktop
Install Claude Desktop from anthropic.com. Available for macOS and Windows. This is the host application that manages your MCP connections.
2
Open the config file
The config file location depends on your OS. If the file doesn't exist yet, create it.

macOS ~/Library/Application Support/Claude/claude_desktop_config.json Windows %APPDATA%\Claude\claude_desktop_config.json
3
Add a server entry
Paste the following into your config file. Replace /Users/yourname/Documents with an actual folder path on your machine. { "mcpServers": { "filesystem": { "command": "npx", "args": [ "-y", "@modelcontextprotocol/server-filesystem", "/Users/yourname/Documents" ] } } }
4
Restart Claude Desktop
Fully quit and relaunch Claude Desktop. The config file is only read on startup — changes won't take effect until you restart.
5
Verify it's working
Open a new conversation and type: "List the files in my Documents folder." Claude should call the list_directory tool and return the results. You'll see a tool-use indicator appear in the chat.
Section 07
Code Examples
A minimal MCP server with one tool: search_web(query). It calls the Brave Search API and returns the top five results. Pick your language — the structure is identical in both.
Install: pip install mcp httpx
import httpx from mcp.server.fastmcp import FastMCP mcp = FastMCP("brave-search") BRAVE_API_KEY = "your-api-key-here" @mcp.tool() def search_web(query: str) -> str: """Search the web using Brave Search and return top 5 results.""" response = httpx.get( "https://api.search.brave.com/res/v1/web/search", headers={"X-Subscription-Token": BRAVE_API_KEY}, params={"q": query, "count": 5}, ) response.raise_for_status() results = response.json().get("web", {}).get("results", []) return "\n\n".join( f"{r['title']}\n{r['url']}\n{r.get('description', '')}" for r in results ) if __name__ == "__main__": mcp.run()
Install: npm install @modelcontextprotocol/sdk zod
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js"; import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js"; import { z } from "zod"; const BRAVE_API_KEY = "your-api-key-here"; const server = new McpServer({ name: "brave-search", version: "1.0.0" }); server.tool( "search_web", "Search the web using Brave Search and return top 5 results.", { query: z.string().describe("The search query") }, async ({ query }) => { const url = new URL("https://api.search.brave.com/res/v1/web/search"); url.searchParams.set("q", query); url.searchParams.set("count", "5"); const res = await fetch(url.toString(), { headers: { "X-Subscription-Token": BRAVE_API_KEY }, }); if (!res.ok) return { content: [{ type: "text", text: `Error: ${res.status} ${res.statusText}` }] }; const data = await res.json(); const results = data?.web?.results ?? []; const text = results .map((r) => `${r.title}\n${r.url}\n${r.description ?? ""}`) .join("\n\n"); return { content: [{ type: "text", text }] }; } ); const transport = new StdioServerTransport(); await server.connect(transport); // requires "type":"module" in package.json
